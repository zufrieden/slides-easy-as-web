First came the hyperlink

Simple as the hyperlink ...
My first program was a Hypercard quizz game I did at school about expo.01
Linking panels was my first sip to interaction design.
Hypercard was a largely inspiring piece of software for Tim Berners Lee to create the web.
Ted Nelson first came with the hypertext concept and Hypercard took it further the web really popularised it.

hyperlink is bi-directional

Link were designed as not bi-directional, you refer someone, but This someone doesn't have to know accept or reject, nor has anything to say about this relationship. When you think about it, this is so much a missing opportunity, so poor comparing to relational databases, proper objects linking. Yet I think this is precisely why web won over other technologies

Proprietary vs Open

Looking at proprietary solutions versus open web technologies

If you think of Macromedia Flash technologies, and before that Java Applets, they tried to fill a missing gap of web features. Web enthousiasts try hard to push web standards and browsers implement thoses gaps

Looking at books refering flash sites from 2005 and realizing how close to web experiences we are now with CSS animations and Javascript.

Yet to come with greater power with Houdini and full implementation of the Web Animation API. Not to mention WebAssembly allowing us to imagine an even more powerfull web. https://ishoudinireadyyet.com/

A rejected promise

I hate obsolescence, and I love theoretical web reslience. HTML, CSS and for a small part Javascript at their core have this rule :

A today's browser can display a 1996 website

A 1996 browser can display a today's webpage

Of course we broke this rule, it is so tempting to use this huge file of Javascript to scrolljack my page. Let's try our best to avoid those situations. And just take anowledgement and write down when we cross this line.

Act 4.
We all love a fast web

Act 4.
We all love a fast web

So fast we forget data has to travel the world

There are tones of performance optimisation to consider when building for the web. Images, Code, Server caching technics, etc.

Today's solutions are tomorrow's problems

HTTP/1.0 all in one CSS file for all your website optimizations are not as fast as HTTP/2.0 having a set of css files for every page types. No silver bullet. We need to test, learn and unlearn when a new implementation come.

Eventually data is already here localy

I enjoy reading and testing offline first solutions. You may have heard of Hoodie which provide an offline first noBackend approach. Offline approach is a great step toward faster web. A more subtle approach to server-client strict model. Service Workers API is now so widely supported there is no excuse to invest time in learning how it work and apply to any website.
I can not recommand enough Jermey Keith's new book "Going Offline". Dive in Service Workers API and explaining man-in-the-middle attacks and strategy to make your website faster for your user. PWA - Progressive Web Apps concept containing
https, ServiceWorkers and the manifest is not about apps only. Every website will benefit from this. There will be abstraction layers to service workers to ease developper's life. As well explained in this book, every website (designwise, contentwise, architecturewise) will implement a different proxy strategy plus the API is already abstracted enough to not use a library. image source : https://twitter.com/adactio/status/986935469178019840

Act 8.
We love an open web

RSS as an exemple

RSS (Really Simple Syndication) is another exemple of simple web standards winning slowly over time. You may think RSS died with Google Reader, or social media took the place as an intermediate syndication platform. Podcasts uses RSS standards for instance. Google trends show a clear decline of the term "RSS" but the reality is more subtle.

For a while, before a third of the planet had signed up for Facebook, RSS was simply how many people stayed abreast of news on the internet.

Aaron Swartz – The Rise and Demise of RSS
Swartz helped a lot to get RSS extensible. The working group standardize an open standard allowing the way to podcast for instance.

Designed to embrace new usages

The way RSS was designed (read a full story to understand its origins an forks) shows how open standards allow innovations to grow
